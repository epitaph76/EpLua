Подбор модели LLM для генерации кода

Для задачи генерации Lua-кода на локальной лёгкой модели лучшим выбором будут открытые кодовые LLM с небольшим размером. Так, модель Qwen 2.5 Coder 3B (4-битная квантизация, контекст 4096) показала выдающиеся результаты на бенчмарках по генерации кода и остаётся достаточно «лёгкой» для 8 GB GPU. Альтернативой может стать Phi-3.5/4 Mini (3.8B) – она заточена на длинный контекст и хорошо работает с кодом, хотя при этом требует около 6–10 GB VRAM в квантизированном виде. При необходимости можно попробовать вариант Qwen 2.5 Coder 7B с Q4-квантизацией (около 6–7 GB), но 3B-версия безопасна для заданных ограничений. В качестве запасного варианта подойдут и другие малые кодовые модели (например, CodeLlama 7B или Llama 3.2 3B), однако именно Qwen Coder и Phi Mini лидируют по качеству генерации при 8 GB.
Идеи из Qwen Code и Claw Code

Для проектирования агентской системы полезно учесть архитектурные решения открытых фреймворков Qwen Code и Claw Code. Qwen Code – это open-source CLI‑агент для кода, оптимизированный под серию моделей Qwen. Он предоставляет готовый набор «навыков» (skills) и субагентов для анализа и генерации кода, реализуя агентный workflow и инструментальную архитектуру (интеграция с IDE, команды анализа кода и пр.). Claw Code – открытая пере-реализация Claude Code на Python+Rust, поддерживающая произвольные локальные LLM. Он воспроизводит модульную систему инструментов (файловые операции, веб-поиск, запуск кода и т.д.), а также управление памятью и многопоточную архитектуру агентов. Главное, что даёт Qwen и Claw – это идея агентности: агент может сам задавать уточняющие вопросы, запускать подзадачи и итеративно улучшать код в несколько шагов. Мы можем использовать это как образец: создавать для нашего агента «skills», такие как проверка синтаксиса Lua, запуск тестов, отладка – и давать модели возможность повторять генерацию, пока результаты не станут корректными.
Архитектура агентской системы и pipeline

Предлагаемый пайплайн работы такой: пользователь формулирует задачу на русском или английском, система (LLM-агент) генерирует первоначальный Lua-скрипт, затем автоматически проверяет его. Проверка выполняется отдельными инструментами: например, сначала lua -p (парсер) или luacheck для синтаксического анализа, затем (при возможности) запускаем скрипт на тестовых данных. Если в коде найдены ошибки или неверные результаты, агент получает это как обратную связь и повторно дорабатывает код, при необходимости задавая уточняющие вопросы пользователю. Такая многошаговая обработка напоминает Конвейер CI/CD: сначала статический анализ (линтинг, типизация), потом тесты, и только потом финальная выдача. Аналогично, можно реализовать «валидационный конвейер» – например, при первой генерации выполнять luac -p (или запуск в защищённой виртуальной среде), фиксировать ошибки и передавать их в LLM для исправления. Вдохновляясь Claw Code, можно спроектировать систему инструментов (Tools) – каждый выполняет конкретную задачу (проверка синтаксиса, отладочный запуск, паттерн-фильтрация). Эта схема гарантирует, что генерация – не одноразовый вывод, а цикл: LLM генерирует, валидатор анализирует, LLM переделывает.
Проверка и валидация кода

Код должен проверяться любым возможным способом на корректность. Синтаксис Lua мы проверяем прямо интерпретатором (lua -p), или статическим анализатором Luacheck. Логические ошибки можно улавливать через набор тестов или через статические проверки шаблонов (например, поиск типичных ошибок Lua). Этот этап действует как «шлюз качества»: похожий на CI-процесс, он включает линтинг, юнит-тесты и сборку. Например, можно автоматически прогонять скрипт с заранее известными входными данными (из примеров задачи) и сравнивать с ожидаемым выводом – при несовпадении возвращаем ошибку агенту. Если тесты не пройдены, система передаёт моделям сообщение вида «Ошибка: при запуске скрипта на входе X получил Y вместо Z», и агент исправляет код. Таким образом, цикл поколение→проверка→уточнение повторяется, пока код не станет работоспособным и соответствующим заданию.
Технический стек и реализация

Мы используем Python как основной язык для серверной логики и оркестрации агентов (Claw Code частично на Python, что удобно). Инференс LLM организуем через Ollama: в README следует указать точную команду ollama pull, например ollama pull qwen2.5-coder:3b, и параметры запуска ollama run c num_ctx=4096, num_predict=256, batch=1, parallel=1 на GPU с квантованием Q4_K_M. Для взаимодействия с моделью можно запустить Ollama как API или через вызовы командной строки.

Компоненты:

    ML-модель и Ollama: Qwen 2.5 Coder (3B) или Phi Mini. Указанный образец ollama pull qwen2.5-coder:3b потребляет ~2–3 GB памяти, что укладывается в 8 GB.
    API-сервер: FastAPI (или Flask) с реализацией контракта из localscript-openapi.zip. Он принимает запросы, запускает генерацию и валидацию, и возвращает Lua-код по спецификации.
    Инструменты валидации: системные вызовы lua/luac, библиотека Lupa (вызов интерпретатора Lua из Python) или готовые пакеты для проверки Lua.
    Хранение знаний: при необходимости можно включить локальную базу знаний – например, SQLite или файлы с шаблонами решений, по которым агент проводит поиск (RAG). Все ресурсы должны быть локальными и открытыми.
    Контейнеризация: Docker (с поддержкой NVIDIA GPU) для всего стека: Python-приложение, Ollama (с GPU-драйвером), инструменты Lua. В Dockerfile необходимо указать установку зависимостей, загрузку моделей (ollama pull ...), и одно-командный запуск всего через docker-compose up.

Интерфейс (UI)

Хотя UI не обязателен, дополнительный веб-интерфейс улучшит демо. Можно использовать готовую Open WebUI (открытый чёртов интерфейс для локальных LLM) – он поддерживает подключение к Ollama и предоставляет чат-стиль UI. В таком случае фронтенд OpenWebUI остаётся почти без изменений, его нужно лишь настроить на нашу Ollama-инстанцию (документация показывает, что Open WebUI автоматически соединяется с Ollama через API). Как вариант, можно форкнуть OpenWebUI, удалить из него ненужный бекенд и подключить собственные эндпоинты (и Ollama) – либо написать простой интерфейс на React/Flask, где пользователь вводит задачу и видит итоговый Lua-код. Этот интерфейс может включать историю итераций: исходный запрос, ответы агента, вопросы уточнения, финальный код.

Итого: предложенное решение объединяет лёгкую локальную LLM (Qwen 2.5 Coder) с агентной архитектурой в духе Qwen Code/Claw Code, многоступенчатую проверку кода (линтинг, тесты) и (опционально) веб-интерфейс Open WebUI. Все компоненты локальны и контейнеризированы, что обеспечивает приватность и воспроизводимость. Использование Docker/olv-compose с точными командами (ollama pull, ollama run --num_ctx=4096 ...) позволит жюри быстро развернуть решение.

Источники: Публикации о моделях и инструментах подтверждают выбор модели и архитектуры. Эти материалы иллюстрируют принцип «легкой» кодовой модели на 8 GB и организацию agentic pipeline с инструментами.
