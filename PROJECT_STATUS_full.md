# 2. Что именно строится

Проект строится как **локальная агентная система для генерации LocalScript-совместимого Lua-кода**, а не как обычный чат с LLM и не как general-purpose coding assistant.

Система должна:

* работать **полностью локально**;
* использовать **открытую модель**, запускаемую через **Ollama**;
* укладываться в **8 GB VRAM** без CPU offload;
* выдавать не “абстрактный Lua”, а **корректный ответ под конкретный LocalScript-контур**;
* уметь не только генерировать, но и **проверять** и **чинить** ответ;
* быть **воспроизводимой** через Docker и документированный запуск;
* иметь хотя бы **один полезный agentic uplift**, а не просто single-shot вызов модели.

# 3. Откуда берутся архитектурные идеи

Здесь важно говорить прямо. Проект не “вдохновляется агентностью вообще”, а использует конкретные идеи из двух open-source code-agent систем.

## 3.1. Что берём из Qwen Code

Из **Qwen Code** в проект напрямую берутся следующие инженерные идеи:

1. **Code-agent workflow**
   Не один вызов модели, а последовательность шагов:

   * анализ задачи;
   * выбор режима;
   * генерация;
   * проверка;
   * исправление.

2. **Skills / action decomposition**
   Агент работает не как один огромный промпт, а как набор прикладных действий.
   В нашем проекте эти действия превращаются в:

   * классификацию задачи;
   * выбор archetype;
   * выбор output mode;
   * генерацию Lua;
   * формирование уточняющего вопроса;
   * repair после ошибок.

3. **Code-first подход**
   Центр системы — не чат, а инженерный результат. Пользователь должен получать рабочий код в нужном формате, а не длинный разговор.

4. **Разделение шага генерации и шага критики**
   Даже маленькая модель работает лучше, если generation и review/repair разведены по этапам.

## 3.2. Что берём из Claw Code

Из **Claw Code** в проект напрямую берутся следующие инженерные идеи:

1. **Tool-based execution architecture**
   Проверка и улучшение кода должны выполняться не “магией модели”, а конкретными инструментами и проверками:

   * format validator;
   * path validator;
   * forbidden-patterns checker;
   * syntax checker;
   * benchmark runner;
   * retrieval selector;
   * repair executor.

2. **Оркестрация как state machine**
   Агентность должна быть описана как набор состояний и переходов, а не как расплывчатое “модель сама решит, что делать дальше”.

3. **Модульное разделение компонентов**
   Нельзя смешивать:

   * backend API;
   * model adapter;
   * orchestration;
   * validators;
   * retrieval;
   * benchmark harness.

4. **Расширяемость без переписывания ядра**
   Даже если на старте делается минимальная версия, архитектура должна позволять добавлять retrieval, UI и benchmark layer без развала core pipeline.

## 3.3. Что НЕ копируем из Qwen Code и Claw Code

Мы не строим полную локальную копию Claude Code, Qwen Code или универсального coding agent.

В проект **не входят**:

* general-purpose autonomous coding agent;
* интернет-доступ;
* тяжёлый zoo инструментов;
* IDE-платформа;
* сложная память ради памяти;
* “агентность ради агентности”;
* многопоточная архитектура без доказанной пользы для кейса.

Наша цель — **узкий, воспроизводимый, локальный Lua/LocalScript-агент**.

# 4. Архитектурная формула проекта

Проект состоит из следующих слоёв:

1. **Input Layer**
   Принимает задачу на русском или английском, а при необходимости — контекст, JSON и/или существующий код.

2. **Task Understanding Layer**
   Классифицирует задачу, определяет archetype и формат ожидаемого ответа.

3. **Model Invocation Layer**
   Вызывает локальную модель через Ollama.

4. **Domain Adapter Layer**
   Принуждает генерацию к требованиям LocalScript:

   * допустимые пути данных;
   * нужный output mode;
   * допустимая структура ответа;
   * корректное использование `wf.vars` и `wf.initVariables`.

5. **Validation Layer**
   Проверяет ответ на форматные, структурные и частично семантические ошибки.

6. **Critic & Repair Layer**
   Если ответ не проходит — формирует точечную задачу на исправление и запускает ограниченный repair loop.

7. **Output Layer**
   Возвращает финальный ответ, trace шагов и статус проверки.

# 5. Общие ограничения проекта

## 5.1. Технические ограничения

* только локальный inference;
* только open-source / локально доступные модели;
* запуск через Ollama;
* бюджет GPU: **8 GB VRAM**;
* без внешних AI API;
* без зависимости от облачного inference runtime;
* воспроизводимость обязательна.

## 5.2. Ограничения предметной области

Система должна учитывать, что задача — это не “любой Lua”, а **LocalScript-специфичный Lua**.

Критично соблюдать:

* допустимые пути данных;
* корректную работу с `wf.vars.*`;
* корректную работу с `wf.initVariables.*`;
* запрет на недопустимые паттерны доступа;
* различие между:

  * raw Lua,
  * JSON-with-lua-wrapper,
  * patch existing code.

## 5.3. Ограничения архитектуры

* никакой тяжёлой автономии;
* repair loop обязан быть ограничен;
* UI не должен становиться центром проекта;
* retrieval допускается только если он локален и даёт измеримый прирост;
* все части системы должны быть воспроизводимы с нуля.

Разобрать предметную область так, чтобы модель и агентный pipeline оптимизировались под **конкретные классы задач LocalScript**, а не под “Lua вообще”.

* описываем допустимые пути данных;
* классифицируем выходные форматы;
* собираем initial regression pack;
* составляем список типовых ошибок маленькой модели.

### Какие классы задач выделяем

| Класс              | Что означает                 | Ключевая сложность                 |
| ------------------ | ---------------------------- | ---------------------------------- |
| Простое извлечение | чтение значения по пути      | точный доступ к данным             |

| Patch mode         | доработка существующего кода | нельзя переписать всё заново       |
| JSON wrapper mode  | код внутри JSON-поля         | корректный wrapper и экранирование |

## S-2. Финальный отбор модели под 8 GB VRAM

**Статус:** `deferred`

### Временное решение

До возврата к этому этапу проект использует provisional model tag `qwen2.5-coder:3b`.

Этот tag нужен как рабочая база для проектирования agent contour, validators и repair loop, но **не считается финальным выбором модели**.

### Цель

Выбрать финальную локальную модель, которая проходит ограничения жюри и даёт максимальное качество в целевом домене **внутри реального agent pipeline**, а не по isolated single-shot benchmark.

### Зависимости

* завершён S-1;
* спроектирован agent contour;
* реализованы domain adapter и validator / repair loop;
* собран initial benchmark pack.

### Откуда берутся идеи

* Из Qwen Code берётся ориентация на **кодовую модель, а не general chat model**.
* Из Claw Code берётся инженерный принцип: модель выбирается по **измеримому поведению в рабочем пайплайне**, а не по репутации.

### Что должно быть на входе

* benchmark pack;
* agent contour;
* validation / repair loop;
* обязательные runtime-параметры;
* понятные критерии ошибки:

  * неверный код,
  * неверный формат,
  * неверные пути к данным,
  * лишние конструкции.

### Что делаем

* собираем shortlist кандидатов;
* тестируем реальные Ollama tags внутри одного и того же agent pipeline;
* сравниваем квантования;
* меряем peak VRAM;
* меряем latency;
* меряем pass@1;
* меряем pass after repair;
* отдельно смотрим RU/EN запросы;
* фиксируем слабые места кандидатов.

### Основной shortlist

## S-3. Выделение агентного контура из Qwen Code и Claw Code

**Статус:** `done`

### Цель

Спроектировать минимальную агентную схему, которая реально повышает качество, но не превращает проект в тяжёлого автономного code-agent’а.

### Зависимости

* завершён S-1;
* зафиксирован provisional model tag `qwen2.5-coder:3b`.

### Откуда берутся идеи

Здесь источники должны быть указаны прямо:

| Архитектурная идея               | Источник  | Как используется                                           |
| -------------------------------- | --------- | ---------------------------------------------------------- |
| Многошаговый code-agent workflow | Qwen Code | classify → generate → validate → repair                    |
| Skill decomposition              | Qwen Code | classify, choose mode, generate, review, ask clarification |
| Tool-based execution             | Claw Code | validators и checks как отдельные инструменты              |
| State-machine orchestration      | Claw Code | пайплайн как набор явных состояний                         |
| Модульность orchestration        | Claw Code | разделение orchestrator / model adapter / validators       |

### Что должно быть на входе

* понятен домен;
* зафиксирован provisional model tag;
* известно, какие проверки реально полезны.

### Что делаем

* проектируем agent loop;
* выделяем состояния;
* проектируем stop conditions;
* ограничиваем число repair iteration;
* определяем, когда система:

  * автоматически чинит ответ,
  * задаёт уточняющий вопрос,
  * завершает попытку.

### Предлагаемая state machine

1. `task_classification`
2. `context_check`
3. `mode_selection`
4. `archetype_selection`
5. `generation`
6. `format_validation`
7. `rule_validation`
8. `critic_step`
9. `repair_generation`
10. `finalize`
11. `clarification`

### Таблица состояний

| Состояние             | Вход                 | Выход                              | Условие перехода                              |
| --------------------- | -------------------- | ---------------------------------- | --------------------------------------------- |
| `task_classification` | текст задачи         | archetype                          | когда понятен класс задачи                    |
| `context_check`       | задача + контекст    | достаточно / недостаточно данных   | когда ясно, можно ли решать задачу            |
| `mode_selection`      | archetype + описание | raw / json / patch / clarification | когда выбран формат ответа                    |
| `generation`          | prompt package       | первичный ответ                    | когда модель вернула код                      |
| `format_validation`   | ответ                | pass/fail                          | когда проверен формат                         |
| `rule_validation`     | ответ                | pass/fail                          | когда проверены правила домена                |
| `critic_step`         | ошибки               | repair task                        | когда локализована проблема                   |
| `repair_generation`   | critic feedback      | исправленный ответ                 | когда есть новая версия                       |
| `finalize`            | валидный ответ       | финальный output                   | когда ответ проходит проверки                 |
| `clarification`       | недостаток входа     | вопрос пользователю                | когда без уточнения риск ошибки слишком высок |

### Жёсткие ограничения этапа

* максимум 1–2 repair iteration;
* максимум 1 уточняющий вопрос;
* никакой бесконечной автономии;
* никакого выхода в интернет;
* никакого расширенного tool zoo без пользы.

### Какие артефакты обязаны появиться

* `docs/AGENT_ARCHITECTURE.md`
* `docs/STATE_MACHINE.md`
* `docs/SKILLS.md`
* sequence diagram пайплайна


### Что именно берём из Qwen Code и Claw Code на этом этапе

На этом этапе из **Qwen Code** берётся идея предварительного reasoning перед генерацией: система сначала должна понять задачу, а уже потом писать код.

Из **Claw Code** берётся идея, что reasoning должен быть оформлен как управляемая последовательность состояний, а не как неявный внутренний процесс модели.

Поэтому agent loop проекта строится как reasoning pipeline:
classification → context check → mode selection → archetype selection → generation → validation → critic → repair / clarification.
### Что должно получиться на выходе
Должна появиться простая, конечная и контролируемая agent architecture.

### UX результата

Пользователь должен видеть не хаотичную “болтовню агента”, а управляемый цикл работы.

### Как понять, что этап завершён

Этап завершён, если:

* agent loop описан явно;
* у каждого состояния есть вход и выход;
* stop conditions определены;
* понятно, когда система спрашивает, а когда чинит автоматически.

### Что не входит в этап

* general-purpose coding agent;
* интернет;
* IDE-like execution environment;
* tool zoo ради галочки.

### Что считается провалом этапа

* “агентность” описана только словами;
* не ограничен repair loop;
* не разделены роли generation и critic.

### Риски

* переусложнить агентность;
* сжечь токены и latency без прироста качества;
* перепутать полезную агентность с маркетинговой сложностью.

## S-5. Domain adapter для LocalScript-формата

**Статус:** `done`

### Цель

Построить слой, который принуждает генерацию к LocalScript-специфике и превращает маленькую модель в прикладной инструмент.

### Зависимости

* завершён S-4.

### Откуда берутся идеи

* Из Qwen Code берётся подход к task-specific generation и skill-aware prompting.
* Из Claw Code берётся принцип, что доменная логика должна быть вынесена в отдельный слой, а не размазана по промптам и backend-коду.

### Что должно быть на входе

* формализованы archetypes задач;
* известны output modes;
* собраны примеры корректных ответов;
* backend уже умеет генерировать базовый ответ.

### На какие документы опираемся

* `docs/DOMAIN_MODEL.md`
* `docs/TASK_ARCHETYPES.md`
* `docs/OUTPUT_MODES.md`
* `docs/FORBIDDEN_PATTERNS.md`

### Что делаем

* разводим две независимые оси: `archetype` и `output mode`;
* привязываем каждый archetype к допустимым prompt/template packs;
* выделяем допустимый data root из входа и запрещаем неоснованное смешивание `wf.vars.*` и `wf.initVariables.*`;
* добавляем domain-specific preprocessing для массивов, пустых значений и date/time кейсов;
* принуждаем `raw_lua`, `json_wrapper`, `patch_mode` и `clarification` к разным финальным правилам ответа;
* подмешиваем в prompt package запрещённые паттерны из `docs/FORBIDDEN_PATTERNS.md`;
* нормализуем финальный output и удаляем prose / markdown вне допустимого режима.

### Минимальный контракт domain adapter

На входе adapter получает:

* `task_text`;
* `provided_context`;
* выбранный `archetype`;
* выбранный `output_mode`;
* нормализованный список допустимых data roots;
* risk tags, если они уже известны из benchmark / retrieval слоя.

На выходе adapter обязан вернуть:

* prompt package для генерации;
* ожидаемый формат результата;
* список domain guardrails;
* список forbidden patterns;
* признак, нужен ли `clarification` вместо генерации.

### Режимы ответа

| Режим              | Когда нужен                 | Что обязан сделать adapter            |
| ------------------ | --------------------------- | ------------------------------------- |
| Raw Lua            | нужен чистый Lua-фрагмент   | вернуть только Lua                    |
| JSON wrapper       | нужен JSON со строковым Lua | вернуть валидный JSON и `lua{...}lua` |
| Patch mode         | дан исходный код            | доработать, а не переписать всё       |
| Clarification mode | входных данных не хватает   | вернуть полезный вопрос               |

### Какие артефакты обязаны появиться

* `packages/task-archetypes/`
* `packages/orchestrator/domain_adapter.py`
* `knowledge/templates/`
* набор templates, покрывающий canonical archetypes и canonical output modes
* автопроверки, что `raw_lua`, `json_wrapper`, `patch_mode` и `clarification` не смешиваются между собой

### Что должно получиться на выходе

Система должна уметь:

* стабильно выбирать нужный формат ответа;
* использовать допустимые пути данных;
* избегать запрещённых паттернов;
* поддерживать raw / wrapper / patch modes.

### UX результата

Пользователь должен ощущать, что система понимает именно LocalScript-контур, а не просто “пишет какой-то Lua”.

### Как понять, что этап завершён

Этап завершён, если:

* на `benchmark/public_cases.json` выбор `primary_output_mode` совпадает с разметкой для всех поддержанных кейсов;
* adapter не смешивает `wf.vars.*` и `wf.initVariables.*` без явного основания из входа;
* `raw_lua` возвращает только Lua, `json_wrapper` возвращает валидный JSON c `lua{...}lua`, `patch_mode` возвращает additive payload;
* частота forbidden-pattern нарушений на публичных кейсах ниже, чем у прямого `S-4` baseline без domain adapter.

### Что не входит в этап

* retrieval;
* UI;
* production packaging;
* сложные внешние parser frameworks без нужды.

### Что считается провалом этапа

## S-6. Validator, critic и repair loop

**Статус:** `done`

### Цель

Создать проверяемый слой качества, который повышает надёжность системы даже на малой модели.

### Зависимости

* завершён S-5.

### Откуда берутся идеи

Это этап, где сильнее всего используется наследование от Claw Code:

* проверки должны быть tool-driven;
* critic не должен быть “второй свободной моделью”, а должен преобразовывать нарушения в конкретную repair-задачу;
* repair loop должен быть ограниченным и управляемым.

От Qwen Code здесь берётся идея разнесения generation и review/fix как отдельных фаз работы с кодом.

### Что должно быть на входе

* первичная генерация уже существует;
* известны типовые ошибки;
* известны output modes;
* есть доменные правила.

### На какие документы опираемся

* `docs/STATE_MACHINE.md`
* `docs/AGENT_ARCHITECTURE.md`
* `docs/CONSTRAINTS.md`
* `docs/FORBIDDEN_PATTERNS.md`
* `docs/OUTPUT_MODES.md`

### Что делаем

* строим format validator, который идёт раньше domain rule checks;
* строим wrapper validator для `json_wrapper` и `patch_mode`;
* строим path validator для `wf.*` roots;
* строим forbidden-patterns checker по явному списку из `docs/FORBIDDEN_PATTERNS.md`;
* строим syntax validator для Lua, если runtime позволяет это делать воспроизводимо;
* строим archetype-specific checks по canonical archetypes;
* фиксируем структуру validator report;
* строим critic, который локализует дефект и формулирует repair task;
* строим ограниченный repair loop с budget из `S-3`.

### Минимальная таксономия нарушений

На этом этапе как минимум должны различаться:

* `invalid_json`
* `invalid_wrapper`
* `wrong_data_root`
* `forbidden_pattern`
* `empty_or_garbage_output`
* `lua_syntax_error`
* `archetype_rule_violation`
* `insufficient_input`

### Контракт validator report

Каждый validator возвращает не prose, а структурированный результат с полями:

* `status`
* `failure_class`
* `location`
* `message`
* `repairable`

### Обязательные проверки

| Проверка                       | Что ловит                   | Обязательность                                |
| ------------------------------ | --------------------------- | --------------------------------------------- |
| JSON validator                 | невалидный JSON             | обязательно                                   |
| Wrapper validator              | отсутствующий `lua{...}lua` | обязательно                                   |
| Path validator                 | неверные пути к данным      | обязательно                                   |
| Forbidden patterns checker     | запрещённые конструкции     | обязательно                                   |
| Empty / garbage output checker | мусорный ответ              | обязательно                                   |
| Lua syntax check               | синтаксическая ошибка       | желательно / обязательно при feasible runtime |
| Archetype rule check           | типовые доменные нарушения  | желательно                                    |

### Архитектура repair loop

1. Первая генерация.
2. Валидатор возвращает структурированный список ошибок.
3. Critic преобразует их в краткую задачу на исправление.
4. Модель получает:

   * исходный запрос;
   * проблемный ответ;
   * список нарушений;
   * правила исправления.
5. Генерируется новая версия.
6. Повторная проверка.
7. Либо success, либо clarification/fail.

### Какие артефакты обязаны появиться

* `packages/validators/`
* `packages/orchestrator/critic.py`
* `packages/orchestrator/repair_loop.py`
* структурированная схема validator report в `packages/shared/`
* тестовые кейсы, покрывающие каждый обязательный класс нарушения

### Что должно получиться на выходе

Система должна уметь:

* ловить форматные ошибки;
* ловить доменные ошибки;
* автоматически исправлять часть ответов;
* останавливаться, если исправление нецелесообразно.

### Что реализовано по факту

* добавлены tool-driven validators для `raw_lua`, `json_wrapper`, `patch_mode` и `clarification`;
* validator reports возвращаются как структурированные данные, а не свободный текст;
* `critic` переводит finding в конкретную repair-задачу или clarification/finalize решение;
* repair loop ограничен и соблюдает budget (`repair_count <= 2`, `clarification_count <= 1`);
* для `raw_lua`, `json_wrapper` и `patch_mode` добавлены deterministic repair path для типовых format / wrapper / almost-JSON сбоев;
* debug mode возвращает audit trail: prompt package, raw model calls, validation passes и critic reports;
* полезный repair и clarification path подтверждены тестами и ручными live smoke-прогонами.

### UX результата

Для пользователя это должно ощущаться как “система сначала проверяет себя, а потом отдаёт ответ”.

### Как понять, что этап завершён

Этап завершён, если:

* каждый обязательный validator возвращает структурированный report, а не свободный текст;
* `critic` и `repair_loop` соблюдают budgets из `docs/STATE_MACHINE.md` (`repair_count <= 2`, `clarification_count <= 1`);
* на `benchmark/initial_regression_pack.json` repair uplift измерим и неотрицателен;
* хотя бы один кейс проходит через полезный repair, а хотя бы один ambiguous кейс переводится в `clarification` вместо бесконечного саморемонта.

### Что не входит в этап

* абсолютная гарантированная корректность;
* бесконечный self-healing агент;
* сложный symbolic executor;
* универсальный runtime debugger.

### Что считается провалом этапа

* validator слишком слабый и бесполезный;
* validator слишком жёсткий и ломает нормальные ответы;
* repair loop не ограничен;
* critic просто заново просит “перепиши всё”.

### Риски

* токеножорный цикл;
* высокая latency;
* ложные срабатывания валидатора.

## S-7. Локальная база знаний, шаблоны и retrieval

**Статус:** `planned`

### Цель

Снизить нагрузку на модель и повысить стабильность за счёт локальных примеров, archetypes и шаблонов.

### Зависимости

* завершены S-1 и S-6.

### Откуда берутся идеи

* Из Qwen Code берётся идея, что generation можно усиливать не только промптом, но и подбором более структурированного контекста.
* Из Claw Code берётся принцип модульного tool-like retrieval, встроенного в пайплайн, а не хаотически подмешиваемого.

### Что должно быть на входе

* есть публичные кейсы;
* есть исправленные внутренние кейсы;
* известны archetypes;
* известны частые ошибки модели.

### На какие документы опираемся

* `docs/DOMAIN_MODEL.md`
* `docs/TASK_ARCHETYPES.md`
* `docs/OUTPUT_MODES.md`
* `docs/AGENT_ARCHITECTURE.md`

### Что делаем

* начинаем с file-based retrieval, а не с vector DB;
* собираем библиотеку эталонных примеров в форме, совместимой с `benchmark/public_cases.json`;
* собираем archetype templates и mode-specific format templates;
* индексируем retrieval по `archetype`, `output_mode` и `risk_tags`;
* определяем bounded retrieval budget перед генерацией;
* подключаем retrieved examples к generation и при необходимости к repair, но не в каждый шаг пайплайна.

### Минимальный retrieval contract

Retrieval layer на входе получает:

* `task_text`;
* `archetype`;
* `output_mode`;
* `risk_tags`.

Retrieval layer на выходе возвращает не больше:

* `2` примеров;
* `1` archetype template;
* `1` format/rules pack.

### Сравнение вариантов реализации

| Вариант                       | Плюсы                              | Минусы               | Решение                        |
| ----------------------------- | ---------------------------------- | -------------------- | ------------------------------ |
| Файлы JSON/YAML               | просто и воспроизводимо            | менее гибко          | базовый must-have              |
| SQLite lookup                 | прозрачная структура и локальность | требует ручной схемы | хороший практичный вариант     |
| Локальный embedding retrieval | гибче по похожести                 | сложнее и тяжелее    | только при доказанном выигрыше |
| Большая vector DB             | мощно на бумаге                    | избыточно            | не нужно                       |

### Какие артефакты обязаны появиться

* `knowledge/examples/`
* `knowledge/archetypes/`
* `knowledge/templates/`
* `packages/retrieval/`
* структурированный набор локальных примеров, привязанный к `source_refs` из `benchmark/public_cases.json`

### Что должно получиться на выходе

Система должна уметь перед генерацией получать:

* похожий пример;
* archetype-specific template;
* форматный шаблон;
* правила для типового преобразования.

### UX результата

Пользователь видит не retrieval, а его эффект: меньше галлюцинаций, лучше формат, лучше стабильность на типовых задачах.

### Как понять, что этап завершён

Этап завершён, если:

* retrieval полностью локален и не требует внешних сервисов;
* retrieval context pack воспроизводим и inspectable;
* хотя бы для части archetypes есть измеримый прирост качества без просадки format correctness;
* embedding retrieval не добавляется без доказанного выигрыша относительно file-based baseline.

### Что не входит в этап

* облачные embeddings;
* тяжёлая vector infrastructure;
* retrieval ради галочки.

### Что считается провалом этапа

* retrieval есть, но не даёт прироста;
* retrieval ломает воспроизводимость;
* в few-shot попадает мусор.

### Риски

* переусложнить слой retrieval;
* потерять прозрачность логики;
* подмешивать нерелевантные шаблоны.

## S-9. Evaluation harness и регрессионный набор

* работающий validator;

### На какие документы и данные опираемся

* `docs/CONSTRAINTS.md`
* `docs/DOMAIN_MODEL.md`
* `docs/OUTPUT_MODES.md`
* `benchmark/public_cases.json`
* `benchmark/initial_regression_pack.json`

### Что делаем

* строим benchmark runner для `benchmark/public_cases.json` и `benchmark/initial_regression_pack.json`;

* считаем format correctness;
* считаем domain rule compliance;
* замеряем latency;
* замеряем RU/EN robustness;

### Какие метрики должны считаться

| Метрика            | Значение                                 |
| ------------------ | ---------------------------------------- |

| Domain compliance  | сколько ответов соблюдают правила домена |
| Avg latency        | среднее время ответа                     |
| Repair uplift      | реальный прирост от repair loop          |
| RU/EN robustness   | устойчивость к языку входа               |

### Какие артефакты обязаны появиться

* `packages/benchmark/`
* `benchmark/public_cases.json`
* `benchmark/initial_regression_pack.json`
* машинно-читаемый benchmark report в `benchmark/reports/`
* сводка по метрикам, пригодная для сравнения между версиями

### Что должно получиться на выходе

Команда в любой момент должна уметь ответить:

* какая модель используется;
* каков текущий pass rate;
* помогает ли repair;
* где система ломается;
* не регресснула ли новая версия.

### UX результата

Разработка перестаёт опираться на ощущения.

### Как понять, что этап завершён

Этап завершён, если:

* benchmark runner автоматизирован и прогоняет оба baseline-файла одной командой;
* regression suite существует и привязан к реальным case IDs;
* метрики считаются воспроизводимо;
* можно сравнивать версии системы между собой;
* `pass after repair >= pass@1`;
* format correctness на публичной baseline-выборке не ниже `7/8`.

### Что не входит в этап

* огромный датасет;
* MLOps-платформа;
* leaderboard-сервис.

### Что считается провалом этапа

* качество оценивается вручную;
* нет regression suite;
* улучшили один кейс и сломали остальные.

### Риски

* переоценить ручные впечатления;
* не хранить негативные кейсы;
* забыть про форматную корректность как отдельную метрику.

### На какие документы и данные опираемся

* `benchmark/public_cases.json`
* `benchmark/initial_regression_pack.json`
* финальный benchmark report
* `README.md`
* demo video / slides

### Что делаем

* делаем последний regression run;
* донастраиваем prompt и repair policy;
* убираем лишнее;
* готовим ответы на вопросы жюри;
* собираем demo script;
* готовим fallback story.

### Минимальный набор финальных прогонов

Перед защитой обязательно прогоняются:

* smoke path (`/health` + один `POST /generate`);
* `benchmark/public_cases.json`;
* `benchmark/initial_regression_pack.json`;
* сценарии, которые идут в demo script.

### Что должно быть в demo script

Demo script обязан содержать:

* открытие проблемы и ограничений;
* один happy-path кейс;
* один кейс с validation / repair или clarification;
* короткий показ локального запуска;
* финальный вывод про measured uplift и ограничения.

### На какие вопросы жюри нужно уметь отвечать

| Вопрос                         | Что нужно показать                  |
| ------------------------------ | ----------------------------------- |
| Почему именно эта модель?      | benchmark + VRAM budget             |
| Почему не single-shot?         | repair uplift и validation layer    |
| Что взято из Qwen Code?        | workflow и skills                   |
| Что взято из Claw Code?        | tool-based checks и state machine   |
| Зачем UI?                      | демо и trace, но не ядро            |
| Почему решение воспроизводимо? | Ollama + Docker + фиксированный tag |

### Какие артефакты обязаны появиться

* финальный regression report
* demo script
* short pitch
* FAQ по защите

### Что должно получиться на выходе

Не просто работающая система, а готовность уверенно объяснить:

* почему эта архитектура выбрана;
* почему она не нарушает ограничения;
* где именно находится agentic uplift;
* как измерялось качество;
* почему решение практично.

### UX результата

Защита должна выглядеть как демонстрация собранной инженерной системы, а не удачного набора промптов.

### Как понять, что этап завершён

Этап завершён, если:

* демо проходит без импровизации минимум на двух сухих прогонах подряд;
* команда знает слабые места;
* защита укладывается во время;
* в системе нет лишних спорных частей;
* после freeze не появилось критичных регрессий в smoke path и baseline benchmark.

### Что не входит в этап

* крупные новые фичи;
* смена модели перед дедлайном;
* переписывание архитектуры на финише.

### Что считается провалом этапа

* поздние хаотичные правки;
* сломанная стабильность;
* команда не может объяснить собственное решение.

### Риски

* перегрузить систему перед сдачей;
* испортить стабильность последними изменениями;
* прийти на защиту без внятной инженерной истории.

# 10. Подробный технический стек

## 10.1. Обязательный стек

| Слой              | Технология                   | Роль                          |
| ----------------- | ---------------------------- | ----------------------------- |
| Inference runtime | Ollama                       | локальный запуск модели       |
| Основная модель   | Qwen 2.5 Coder 3B / Phi Mini | генерация Lua-кода            |
| Backend           | Python + FastAPI             | API и orchestration           |
| Model adapter     | Python module                | вызовы Ollama и постобработка |
| Validation layer  | Python + rule-based checks   | проверка качества             |
| Storage           | файлы / SQLite               | примеры, archetypes, шаблоны  |
| Packaging         | Docker + docker-compose      | воспроизводимость             |

## 10.2. Опциональный стек

| Слой         | Технология                                  | Когда нужен                     |
| ------------ | ------------------------------------------- | ------------------------------- |
| UI           | React / Next.js / Open WebUI adaptation     | если нужен demo/debug интерфейс |
| Retrieval    | SQLite lookup / lightweight local retrieval | если даёт прирост качества      |
| Syntax tools | lua / luac / luacheck / Lupa                | если feasible для validator     |
| Code viewer  | Monaco Editor                               | если делается свой UI           |

## 10.3. Нежелательный или отложенный стек

| Технология                   | Почему нежелательна                   |
| ---------------------------- | ------------------------------------- |
| Cloud AI APIs                | нарушают условия локальности          |
| Большая vector DB            | избыточна для масштаба проекта        |
| Kubernetes                   | не даёт ценности в контексте конкурса |
| Heavy autonomous agent infra | переусложняет решение                 |
| Massive frontend platform    | отвлекает от core pipeline            |

# 12. Главный результат roadmap

Главный результат этого roadmap — не просто локальный вызов маленькой LLM.

Главный результат — появление **узкой прикладной агентной системы для генерации LocalScript-совместимого Lua-кода**, которая:

* работает полностью локально;
* честно помещается в 8 GB VRAM;
* воспроизводима через Ollama и Docker;
* умеет выбирать режим ответа;
* умеет задавать уточнение, когда данных недостаточно;
* умеет проверять и чинить ответ;
* демонстрирует инженерную зрелость, а не только удачный single-shot generation.

# 13. Самый важный принцип проекта

Проект выигрывает не там, где “модель самая умная”.

Проект выигрывает там, где:

* правильно сужен домен;
* жёстко зафиксирован формат ответа;
* выстроен сильный validation layer;
* агентность ограничена и полезна;
* воспроизводимость не ломается;
* UI не подменяет собой ядро решения.
