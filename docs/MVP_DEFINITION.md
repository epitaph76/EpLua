# MVP DEFINITION

Этот документ фиксирует, что будет считаться минимально жизнеспособной версией проекта.

## 1. Цель MVP

MVP должен доказать, что проект способен решать конкурсную задачу как **локальная агентная система**, а не как единичный ответ модели.

## 2. Что MVP обязан уметь

Минимальная рабочая версия обязана:

- принимать задачу на русском или английском языке;
- запускать локальную open-source модель через `Ollama`;
- генерировать LocalScript-совместимый Lua-код;
- поддерживать хотя бы один полезный agentic шаг:
  - repair после ошибки;
  - или clarification при недостатке входных данных;
- выполнять проверяемую валидацию результата;
- возвращать пользователю финальный результат в согласованном формате;
- воспроизводиться по документации без внешних AI API.

## 3. Минимальный состав подсистем MVP

В MVP должны существовать:

- backend API;
- model adapter к `Ollama`;
- task understanding / mode selection на минимально полезном уровне;
- domain adapter под LocalScript-правила;
- validation layer;
- ограниченный critic / repair или clarification loop;
- документированный локальный запуск;
- smoke path для проверки работоспособности.

## 4. Что считается definition of done для MVP

MVP считается готовым, если одновременно выполняются все пункты ниже:

1. Сервис поднимается локально по документированной инструкции.
2. Модель вызывается локально через `Ollama`.
3. Нет использования внешних AI API.
4. Система обрабатывает хотя бы базовый набор публичных сценариев.
5. Ответ генерируется именно в LocalScript-совместимом формате.
6. Есть хотя бы один проверяемый цикл `generation -> validation -> repair/clarification`.
7. Ошибочный или недостаточный ответ не проходит как “успешный” без проверки.
8. README и deployment-документы позволяют воспроизвести запуск на новой машине.

## 5. Что не входит в MVP

В MVP не обязаны входить:

- сложный UI;
- богатый retrieval;
- многомодельная оркестрация;
- продвинутая память агента;
- большой benchmark framework;
- enterprise-grade observability;
- облачная инфраструктура;
- general-purpose coding assistant behavior.

## 6. Какие артефакты должны подтверждать MVP

Когда MVP будет реализован, в репозитории должны появиться как минимум:

- код backend;
- конфигурация model/runtime;
- validator rules;
- минимум один сценарий repair или clarification;
- локальный запуск;
- smoke / regression-проверка;
- обновлённый README;
- deployment-документация.

## 7. Как не спутать MVP с “почти демо”

Следующее **не считается** MVP:

- модель просто отвечает текстом без validation layer;
- система не различает успешный и ошибочный результат;
- запуск зависит от неописанных ручных шагов;
- решение нельзя воспроизвести на новой машине;
- репозиторий содержит красивое описание, но не содержит рабочего пайплайна.

## 8. Текущее состояние

На этапе `S-0` этот документ фиксирует цель MVP, но сам MVP ещё не реализован.
