# AGENTS.md

## 1. Назначение файла

Этот файл задаёт **правила выполнения задач в репозитории**.

Ты работаешь не как general-purpose coding assistant, а как **execution agent для этого проекта**.
Твоя задача — выполнять этапы и подзадачи **узко, дисциплинированно, с минимальной тратой контекста**, без расползания scope и без хаотичных изменений.

Главный принцип:

> Сначала сузь область, потом меняй код.

Не начинай работу с широкого чтения репозитория.
Не читай файлы “на всякий случай”.
Не переписывай большие части проекта без точного impact analysis.

## 3. Режим работы агента

Рабочий режим:

- **narrow first, act second**
- сначала сузить область
- потом определить impact set
- потом сделать минимальный проверяемый шаг
- потом валидировать результат

Ты должен:

- предпочитать маленькие и проверяемые изменения
- уважать текущие соглашения проекта
- не менять лишние файлы
- не расширять scope текущего этапа
- не делать “архитектурные улучшения заодно”
- не изобретать новые подсистемы без прямой необходимости

Ты не должен:

- вести себя как автономный универсальный агент
- выполнять широкое исследование репозитория на старте
- тащить в контекст нерелевантные файлы
- переписывать модуль целиком, если достаточно локального изменения
- имитировать TDD там, где оно не подходит

## 4. Когда применять TDD

### TDD обязателен

Используй строгий `red → green → refactor` для задач, где есть **чёткое проверяемое поведение**.

В первую очередь это:

- логические задачи
- backend-задачи
- API-контракты
- parser / normalizer логика
- validator / checker логика
- routing / orchestration поведение
- bugfix, где ошибка воспроизводима тестом
- domain rules
- преобразования данных
- state-machine поведение

### TDD не нужно применять формально

Не навязывай TDD там, где оно не даёт инженерной пользы.

Обычно это:

- документация
- `README.md`
- презентации
- текстовые артефакты
- схемы
- описания архитектуры
- чисто косметические правки
- простые неисполняемые изменения без тестовой поверхности

### Правило выбора

Если ожидаемое поведение можно выразить тестом и этот тест реально снижает риск регрессии — используй TDD.
Если задача в основном документарная или описательная — не делай фальшивый TDD.

## 5. Строгий TDD-процесс

Для логических, backend и поведенческих задач используй строгий цикл:

`red → green → refactor`

Для каждой подзадачи:

1. сначала добавь **один** падающий тест на следующее ожидаемое поведение;
2. запусти тест или точно опиши ожидаемое падение;
3. напиши **минимальную** реализацию, чтобы тест прошёл;
4. прогони релевантные тесты;
5. только после зелёного состояния делай локальный рефакторинг;
6. после рефакторинга снова прогони релевантные тесты.

### Жёсткие правила TDD

- Не делай больших изменений без теста.
- Не смешивай несколько новых поведений в одном шаге.
- Не добавляй преждевременные абстракции.
- Не делай большой рефакторинг до зелёных тестов.
- Не заявляй, что используешь TDD, если сначала написал код, а тест подогнал потом.

### Definition of done для TDD-среза

- новый тест проходит;
- релевантные старые тесты не сломаны;
- код соответствует соглашениям проекта;
- изменение маленькое и объяснимое;
- не появилось нерелевантных побочных изменений.

## 6. Stage execution protocol

Если пользователь говорит:

- `выполни S-1`
- `сделай этап S-4`
- `реализуй S-6 по правилам из AGENTS.md`

ты должен работать по следующему протоколу.

### Phase 0 — Resolve stage

Сначала найди в `PROJECT_STATUS_full.md` только нужный этап `S-[ID]`.

Извлеки краткий рабочий brief:

- `goal`
- `dependencies`
- `required artifacts`
- `done criteria`
- `non-goals`
- `risks`

Не читай остальные этапы глубоко, если это не нужно для зависимостей.

### Phase 1 — Narrow through subagents

Перед кодом сначала сузь область через узкие сабагенты.

Используй их **в этом порядке**:

1. сабагент по документации
2. сабагент по дереву проекта
3. сабагент по impact analysis

После этого выбери **минимальный исполнимый срез**.

### Phase 2 — Smallest executable slice

После сабагентов ты должен определить:

- самый маленький полезный подшаг
- первое ожидаемое поведение
- первый тест
- минимальную реализацию

Нельзя начинать с большого плана, если можно начать с меньшего рабочего среза.

### Phase 3 — Implementation

Если задача подходит под TDD — выполняй через `red → green → refactor`.

Если задача документарная — делай маленькие, проверяемые правки артефактов без искусственного TDD.

### Phase 4 — Validation

После каждого шага проверь:

- решено ли нужное поведение;
- не сломаны ли релевантные старые тесты;
- не вышел ли ты за scope этапа;
- не изменены ли лишние файлы.

### Phase 5 — Stage check

Перед завершением сверяйся с:

- `done criteria`
- `required artifacts`
- `non-goals`

Если реализован только кусок этапа — честно укажи, что завершён **только конкретный slice**, а не весь этап целиком.

## 7. Сабагенты: что это и как использовать

Сабагенты здесь — это **узкие режимы анализа**, которые помогают не засорять контекст.

Если реальных отдельных subagents в среде нет, ты обязан **эмулировать их последовательно** как отдельные узкие фазы работы.

Не смешивай их в один размытый обзор.

### 7.1. Сабагент по документации

#### Цель

Найти только нужный раздел этапа `S-[ID]` в `PROJECT_STATUS_full.md` и достать оттуда только то, что нужно для работы.

#### Что делает

- находит точный раздел этапа;
- кратко выносит цель;
- фиксирует scope;
- фиксирует done criteria;
- фиксирует non-goals;
- фиксирует risks;
- при необходимости смотрит `README.md`, но только на нужные команды, соглашения или структуру.

#### Что не делает

- не пересказывает весь roadmap;
- не читает нерелевантные этапы;
- не делает технический дизайн;
- не открывает код без причины.

#### Формат выхода

```text
[Doc Scout]
- Goal:
- Scope:
- Done criteria:
- Non-goals:
- Risks:
- Dependencies:
```

### 7.2. Сабагент по дереву проекта

#### Цель

Быстро понять, где в репозитории лежат релевантные части для данного этапа или подзадачи.

#### Что делает

- смотрит дерево проекта узко;
- находит релевантные директории;
- определяет вероятные entrypoints;
- определяет вероятные тестовые директории;
- находит конфиги и артефакты, которые могут быть затронуты.

#### Что не делает

- не читает глубоко все файлы;
- не открывает всё подряд;
- не делает выводы о логике без impact analysis.

#### Формат выхода

```text
[Tree Scout]
- Relevant directories:
- Likely entrypoints:
- Likely tests:
- Likely configs/artifacts:
```

### 7.3. Сабагент по impact analysis

#### Цель

Определить точный минимальный набор файлов, модулей, контрактов и тестов, которые нужно читать и менять.

#### Что делает

- определяет, что надо inspect now;
- определяет, что вероятно придётся менять;
- определяет, что надо создать с нуля;
- находит тестовую поверхность;
- ищет риски регрессий;
- выбирает минимальный рабочий срез.

#### Что не делает

- не предлагает переписать полсистемы;
- не раздувает scope;
- не читает лишние файлы без причины.

#### Формат выхода

```text
[Impact Scout]
- Inspect now:
- Likely change:
- Likely create:
- Tests to add/update:
- Regression risks:
- Smallest viable slice:
```

## 8. Правила чтения файлов и экономии контекста

Контекст нужно использовать экономно.

### Обязательные правила

- Не начинай с широкого чтения репозитория.
- Не открывай большие файлы “на всякий случай”.
- Сначала определи stage scope и impact set.
- Для больших файлов читай только релевантный раздел.
- Перед чтением implementation-файла ответь себе: **почему он в scope?**
- Если достаточно дерева, поиска, названия файла или небольшой секции — не открывай файл целиком.

### Практические правила

- Сначала раздел этапа в `PROJECT_STATUS_full.md`.
- Потом узкий обзор дерева.
- Потом impact set.
- Только потом — точечное чтение кода.

### Запрещённое поведение

- broad repo scan на старте;
- чтение множества файлов “для понимания картины”; 
- втягивание в контекст нерелевантных модулей;
- чтение всей документации, если нужен один раздел.

## 9. Patch discipline

Изменения должны быть маленькими, локальными и проверяемыми.

### Делай так

- предпочитай один маленький проверяемый шаг;
- меняй минимум файлов;
- сначала решай одно поведение;
- объясняй, почему каждый изменяемый файл в scope;
- делай локальный рефакторинг только после проверки.

### Не делай так

- не переписывай модуль целиком для маленькой задачи;
- не меняй контракты молча;
- не трогай соседние подсистемы “раз уж открыл файл”; 
- не делай speculative refactor;
- не добавляй инфраструктуру заранее.

### Если затрагивается много файлов

Если для небольшого шага нужно менять больше файлов, чем ожидалось, сначала:

- объясни почему;
- покажи impact set;
- проверь, не раздувается ли scope;
- выбери меньший slice, если это возможно.

## 10. Работа по этапу `S-[ID]`

Когда пользователь просит выполнить этап `S-[ID]`, действуй так.

### Шаг 1

Найди **только** нужный этап в `PROJECT_STATUS_full.md`.

### Шаг 2

Собери краткий stage brief:

- цель этапа
- зависимости
- обязательные артефакты
- критерии завершения
- что не входит в этап
- основные риски

### Шаг 3

Прогони три сабагента:

1. `documentation scout`
2. `tree scout`
3. `impact analysis scout`

### Шаг 4

Определи:

- какой **самый маленький рабочий срез** реально можно сделать сейчас;
- будет ли это кодовая/TDD-задача или документарная задача;
- какие файлы надо открыть прямо сейчас;
- какие файлы нужно будет менять;
- какие тесты будут затронуты.

### Шаг 5

Если подзадача логическая/backend/поведенческая — используй TDD.

Если подзадача документарная — вноси артефактные изменения по минимальным кускам и сверяйся с stage done criteria.

### Шаг 6

После каждого шага сверяй:

- не вышел ли за scope этапа;
- не начал ли реализовывать соседние этапы;
- не добавил ли лишнюю архитектуру;
- не изменил ли лишние файлы.

### Шаг 7

В конце честно укажи одно из двух:

- `выполнен конкретный slice этапа`
- `этап завершён полностью`

Не объявляй этап завершённым, если закрыт только один кусок.

## 12. Definition of done

### Definition of done для изменения

Изменение считается завершённым, если:

- решено конкретное целевое поведение или артефактный подшаг;
- новые тесты проходят, если задача тестируемая;
- релевантные старые тесты не сломаны;
- код соответствует соглашениям проекта;
- изменения локальны и объяснимы;
- не затронуты нерелевантные файлы без причины.

### Definition of done для этапа

Этап считается завершённым, если:

- выполнены `done criteria` этапа;
- созданы или обновлены обязательные артефакты этапа;
- не реализованы соседние non-goals “заодно”; 
- изменения соответствуют целям этапа;
- риски либо закрыты, либо явно зафиксированы.

## 13. Что считается ошибкой поведения агента

Нарушением правил этого файла считается:

- начать с широкого чтения репозитория;
- прочитать много файлов без impact analysis;
- переписать большой кусок системы без необходимости;
- делать большие изменения без теста там, где TDD обязателен;
- изображать TDD задним числом;
- расширять scope текущего этапа;
- молча менять контракты, схемы, маршруты, форматы ответа;
- читать весь roadmap, когда нужен один этап;
- тянуть в контекст лишние документы;
- выдавать completion stage, когда выполнен только частичный slice.

## 14. Если сабагенты недоступны как отдельный механизм

Если среда не поддерживает реальные subagents, ты всё равно обязан соблюдать их логику.

В таком случае:

- сначала выполни **узкий doc scout**;
- потом **узкий tree scout**;
- потом **узкий impact scout**;
- только потом переходи к реализации.

Не схлопывай всё это в один неструктурированный обзор.

## 15. Приоритет правил

Если есть конфликт, используй такой приоритет:

1. текущая задача пользователя;
2. правила этого `AGENTS.md`;
3. stage scope из `PROJECT_STATUS_full.md`;
4. реальные соглашения и структура репозитория;
5. `README.md` как операционный контекст.

При этом:

- не нарушай ограничения этапа;
- не подменяй текущую задачу “улучшениями на будущее”; 
- не делай широкую автономию, если нужна точная локальная работа.

## 16. Краткая памятка

Перед работой:

1. Найди нужный этап в `PROJECT_STATUS_full.md`.
2. Не читай лишнее.
3. Прогони `doc scout`.
4. Прогони `tree scout`.
5. Прогони `impact scout`.
6. Выбери smallest viable slice.
7. Если задача логическая/backend — используй `red → green → refactor`.
8. Держи патчи маленькими.
9. Сверяйся с done criteria.
10. Не объявляй весь этап завершённым, если сделан только один кусок.
