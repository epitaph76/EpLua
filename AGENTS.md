# AGENTS.md

## 1. Назначение

Этот файл задаёт общие правила работы агента в репозитории.

Агент работает как **execution agent**:

- сначала сужает область задачи;
- потом определяет минимальный impact set;
- потом делает маленькое проверяемое изменение;
- потом валидирует результат.

Главный принцип:

> Narrow first, act second.

---

## 2. Нулевой шаг

Если пользователь в запросе ссылается на `AGENTS.md`, первое действие агента:

1. открыть локальный `AGENTS.md`;
2. прочитать его;
3. только после этого переходить к другим файлам и действиям.

До чтения локального `AGENTS.md` нельзя:

- читать код;
- сканировать дерево проекта;
- открывать документацию "на всякий случай";
- начинать impact analysis.

---

## 3. Режим работы

Агент должен:

- предпочитать маленькие и проверяемые шаги;
- менять минимум файлов;
- уважать текущие соглашения проекта;
- держать scope узким;
- объяснять, почему файл попал в scope;
- валидировать результат после каждого значимого шага.

Агент не должен:

- начинать с широкого исследования репозитория;
- тянуть в контекст нерелевантные файлы;
- переписывать модуль целиком, если хватает локального изменения;
- делать "архитектурные улучшения заодно";
- расширять задачу без прямого запроса.

---

## 4. Источники истины

Если в проекте есть документы, определяющие этапы, требования, ограничения, запуск или структуру, считай их источниками истины.

Обычно это:

- roadmap / status / plan документы;
- `README.md`;
- `docs/*`;
- спецификации, ADR, task-файлы;
- issue, если пользователь явно работает по нему.

Правила использования:

- читай только релевантный раздел;
- не открывай большой документ целиком без необходимости;
- если задача ссылается на этап, milestone, phase или section, найди только нужный фрагмент;
- если явного документа-источника нет, честно зафиксируй это и опирайся на структуру репозитория и формулировку пользователя.

---

## 5. Сужение области

Перед реализацией агент обязан пройти три узкие фазы:

1. `doc scout`
2. `tree scout`
3. `impact scout`

Если среда поддерживает реальные сабагенты, используй их для этих фаз.
Главный агент должен получать от них только краткие выжимки.

Если реальных сабагентов нет, выполни те же фазы последовательно сам, не схлопывая их в один размытый обзор.

### Doc Scout

Цель:

- найти только нужные требования, ограничения и критерии завершения;
- не читать лишнюю документацию.

Формат:

```text
[Doc Scout]
- Goal:
- Scope:
- Done criteria:
- Non-goals:
- Risks:
- Dependencies:
```

### Tree Scout

Цель:

- быстро понять, где лежат релевантные части проекта.

Формат:

```text
[Tree Scout]
- Relevant directories:
- Likely entrypoints:
- Likely tests:
- Likely configs/artifacts:
```

### Impact Scout

Цель:

- определить минимальный набор файлов, которые нужно читать, менять или создать.

Формат:

```text
[Impact Scout]
- Inspect now:
- Likely change:
- Likely create:
- Tests to add/update:
- Regression risks:
- Smallest viable slice:
```

---

## 6. Smallest Viable Slice

После фаз сужения области агент должен определить:

- самый маленький полезный подшаг;
- первое ожидаемое поведение или артефакт;
- минимальный набор файлов для текущего шага;
- способ проверки результата.

Нельзя начинать с большого плана, если можно начать с меньшего рабочего среза.

---

## 7. Когда применять TDD

Используй строгий `red -> green -> refactor`, если задача описывает проверяемое поведение и тест реально снижает риск регрессии.

Обычно TDD обязателен для:

- логики;
- backend-поведения;
- API-контрактов;
- parser / normalizer / validator логики;
- orchestration / routing;
- bugfix, который воспроизводится тестом;
- domain rules;
- state-machine поведения.

Не навязывай TDD для:

- документации;
- текстовых артефактов;
- схем и описаний;
- чисто косметических правок;
- изменений без нормальной тестовой поверхности.

---

## 8. Строгий TDD-цикл

Для поведенческих задач работай так:

1. добавь один падающий тест на следующее поведение;
2. зафиксируй ожидаемое падение;
3. напиши минимальную реализацию;
4. прогони релевантные тесты;
5. только после зелёного состояния делай локальный рефакторинг;
6. после рефакторинга снова прогони релевантные тесты.

Жёсткие правила:

- не делай большие изменения без теста там, где TDD уместен;
- не смешивай несколько новых поведений в одном шаге;
- не делай большой рефакторинг до зелёного теста;
- не заявляй TDD задним числом.

Формат:

```text
[TDD Step - Red]
- Behavior:
- Test:
- Expected failure:

[TDD Step - Green]
- Minimal implementation:
- Files touched:

[TDD Step - Verification]
- Tests run:
- Result:

[TDD Step - Refactor]
- Refactor done/skipped:
- Reason:
```

---

## 9. Документарные изменения

Если задача документарная, не имитируй TDD.

Работай маленькими, проверяемыми правками:

- измени только нужный артефакт;
- сверь изменение с целью задачи;
- не переписывай большие куски текста без причины;
- оставь краткую фиксацию, что именно изменено и что ещё осталось.

Формат:

```text
[Artifact Step]
- Artifact:
- Goal of change:
- Files touched:
- Validation:
- Remaining gaps:
```

---

## 10. Patch Discipline

Изменения должны быть маленькими, локальными и объяснимыми.

Делай так:

- меняй минимум файлов;
- сначала решай одно поведение или один артефактный подшаг;
- перед чтением файла понимай, почему он в scope;
- после валидации делай только локальный рефакторинг.

Не делай так:

- не переписывай полсистемы ради маленькой задачи;
- не меняй контракты молча;
- не трогай соседние подсистемы "раз уж открыл";
- не добавляй инфраструктуру заранее;
- не делай speculative refactor.

Если impact set внезапно разрастается, сначала:

- объясни почему;
- покажи новый impact set;
- проверь, нельзя ли выбрать меньший slice.

---

## 11. Системность решений

Решения должны быть системными в пределах заявленного scope.

Обязательные правила:

- нельзя хардкодить поведение под один тест, один пример, один вход или одну частную задачу;
- нельзя "подгонять" реализацию под текущий кейс, если ожидается повторяемая логика;
- если задачу можно решить общим правилом, конфигурацией, нормальной веткой логики или расширением существующего механизма, нужно предпочесть это;
- если допустимо только одноразовое частное решение, это должно быть прямо продиктовано задачей пользователя, типом артефакта или природой работы, например для миграции, ad hoc-скрипта или временного фикса.

В сомнительных случаях агент должен выбирать не самый быстрый хардкод, а минимальное системное решение без раздувания scope.

---

## 12. Проверка результата

Изменение считается завершённым, если:

- решён конкретный целевой подшаг;
- новые тесты проходят, если задача тестируемая;
- релевантные старые тесты не сломаны;
- решение не захардкожено под один частный кейс, если задача не была одноразовой;
- изменения локальны и объяснимы;
- не затронуты нерелевантные файлы.

Если закрыт только кусок большой задачи, так и пиши:

- `выполнен конкретный slice`

Если действительно закрыты все заявленные критерии:

- `задача завершена полностью`

---

## 13. Ошибки поведения агента

Нарушением правил считается:

- не открыть локальный `AGENTS.md` первым действием, когда пользователь на него ссылается;
- начать с широкого чтения репозитория;
- читать много файлов без узкого impact analysis;
- переписывать большой кусок системы без необходимости;
- хардкодить решение под один тест, один пример или один кейс там, где нужна повторяемая логика;
- делать большие изменения без теста там, где TDD обязателен;
- изображать TDD задним числом;
- молча менять контракты и форматы;
- тянуть в контекст лишние документы;
- объявлять всю задачу завершённой, когда закрыт только частичный slice.

---

## 14. Приоритет правил

Если есть конфликт, используй такой порядок:

1. текущая задача пользователя;
2. правила этого `AGENTS.md`;
3. явные требования из релевантных проектных документов;
4. реальные соглашения и структура репозитория.

---

## 15. Краткая памятка

Перед работой:

1. если пользователь сослался на `AGENTS.md`, открой его первым;
2. сузь область через `doc scout`, `tree scout`, `impact scout`;
3. выбери smallest viable slice;
4. выбери минимальное системное решение, а не хардкод под один кейс;
5. если задача поведенческая, используй `red -> green -> refactor`;
6. держи патчи маленькими;
7. валидируй результат;
8. не объявляй всю задачу завершённой, если сделан только кусок.
