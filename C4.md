# C4 Architecture Description for `luaMTS`

## Scope and reading rules

This document is a textual C4 description of the current `luaMTS` repository state.

It is based on:

- `README.md`
- `docker-compose.yml`
- `docs/AGENT_ARCHITECTURE.md`
- `docs/AGENT_PIPELINE_SEQUENCE.md`
- `docs/DOMAIN_MODEL.md`
- actual runtime code in `apps/api/*` and `packages/*`

Important interpretation rules:

1. This document describes the current implementation first.
2. If a directory exists only as a placeholder, it is marked as inactive.
3. If a module exists as an extensibility point but is not on the primary runtime path, it is marked as secondary.
4. `luaMTS` is a local-first system by design. Cloud AI APIs are outside the intended architecture.

The goal of the system is narrow and explicit:

- accept a user task plus optional JSON context;
- generate LocalScript-compatible Lua for the LowCode contract;
- validate the candidate through deterministic checks;
- perform a bounded repair when possible;
- expose the workflow through a local API and CLI;
- remain reproducible through Docker and a local Ollama runtime.

## C1. System Context

### System under consideration

`luaMTS` is a local agentic generation system for LowCode or LocalScript-compatible Lua.

The system is not a general-purpose coding assistant and is not designed as an internet-connected autonomous platform. Its boundary is intentionally narrow:

- local user interaction;
- local HTTP API;
- local CLI;
- local LLM runtime through Ollama;
- local deterministic validation and repair;
- local benchmark and reporting assets.

### Primary context statement

In C4 terms, `luaMTS` is the software system that sits between a human operator and a local language model runtime. It takes natural-language task descriptions plus structured context, converts them into a constrained generation workflow, evaluates outputs against the project-specific contract, and returns either a valid result, a repaired result, or structured failure information.

### External actors and external systems

| Type | Name | Description | Why it is outside the `luaMTS` logical boundary |
| --- | --- | --- | --- |
| Person | End User / Operator | Human who provides a task, optional JSON context, clarifications, and feedback. | The user consumes the system and is not implemented by the repository. |
| Person | Developer / QA / Benchmark Operator | Human who runs tests, benchmarks, smoke checks, Docker startup, and regression analysis. | This role operates the solution but is not part of the runtime logic. |
| Software System | Local Ollama Runtime | Local model server used through `/api/generate` and `/api/chat`. | `luaMTS` depends on it but does not implement the model runtime itself. |
| Platform | Local OS / Docker Engine | Hosts Python runtime, binaries, containers, files, and ports. | Execution platform, not business logic. |

### System context relationships

| Source | Relationship | Target | Intent |
| --- | --- | --- | --- |
| End User / Operator | submits tasks and context through CLI or API | `luaMTS` | Main business interaction. |
| End User / Operator | receives code, validation status, clarification questions, progress, and repair summary | `luaMTS` | Main output channel. |
| Developer / QA / Benchmark Operator | runs tests, benchmark scripts, Docker startup, and diagnostics | `luaMTS` | Engineering and evaluation interaction. |
| `luaMTS` | sends prompt requests to local inference endpoints | Local Ollama Runtime | LLM-backed planner, prompter, generator, and semantic critic layers. |
| `luaMTS` | reads and writes local files, datasets, prompt templates, reports, and temporary artifacts | Local OS / Docker Engine | Local persistence and execution substrate. |

### Narrative of the context

At the highest level, the system works as a local translation and quality gate between a human request and a local language model:

1. The user provides a task in Russian or English.
2. The system optionally narrows the input context and infers data roots such as `wf.vars.*` or `wf.initVariables.*`.
3. The system invokes planner and prompter agent layers to shape the request.
4. The system invokes a generator through Ollama.
5. The system validates the output against LowCode and Lua constraints.
6. If the candidate is repairable and budget remains, the system performs a bounded repair loop.
7. The system returns a final response with trace and validation metadata.

### Architectural boundaries visible already at C1

- No internet-facing dependencies are required for normal runtime generation.
- No external AI APIs are part of the intended architecture.
- Ollama is local and guarded by policy checks that reject non-local base URLs.
- The system is intentionally domain-narrow: it optimizes for LowCode Lua generation, not arbitrary coding.

### Container narrative

#### 1. CLI Client

The CLI is a first-class client, not just a thin debug script.

It supports:

- interactive chat mode;
- one-shot generation;
- `/plan` one-shot planning mode;
- feedback and assisted-repair interaction flows;
- API-backed and direct-Ollama execution modes;
- release, releaseSlim, and debug runtime modes;
- report export;
- VRAM checks and doctor commands.

Architecturally, the CLI plays two roles:

- a user-facing interface to the backend API;
- a controlled bypass path for diagnostics, where validation is intentionally skipped and the trace becomes `direct_ollama`.

#### 2. HTTP API Backend

The API backend is the core application container.

It owns:

- request validation and response shaping;
- runtime mode policy;
- planner and prompter orchestration;
- prompt rendering;
- generator calls;
- deterministic validation;
- semantic critique;
- bounded repair loop;
- progress streaming;
- debug trace generation.

This is the central runtime container of the system.

#### 3. Knowledge and Prompt Assets

This is a file-based data container, not a database.

It stores:

- archetype-specific JSON templates;
- common and mode-specific prompt rules;
- example corpora;
- future feedback capture directories.

Current-state nuance:

- prompt templates are active;
- example retrieval infrastructure exists as code and data;
- main LowCode prompt builder currently initializes an empty `RetrievalPack`, so retrieval is present architecturally but not yet materially enriching the primary prompt path;
- `knowledge/feedback/` is explicitly a future placeholder and is not used by runtime code.

#### 4. Benchmark Runner

The benchmark runner is an engineering container, not a user-facing runtime container.

It:

- loads datasets;
- constructs requests;
- instantiates `GenerationService`;
- runs generation with validation and repair;
- evaluates candidates with benchmark principles;
- writes cumulative reports after each case.

This makes benchmark execution tightly aligned with the same service logic used by the application, which improves architectural consistency.

#### 5. Dataset and Artifacts Stores

These are repository-local stores.

The benchmark datasets are inputs to evaluation.
The artifact store is the output sink for reports and run traces.

They are local by design and support reproducibility.

#### 6. Optional UI Placeholder

The `apps/ui` directory currently contains only `.gitkeep`.

For C4 purposes, this is not an implemented container.
It is a reserved slot in the repository layout.

### Deployment note

The deployment topology in `docker-compose.yml` is worth stating separately:

- `ollama` is the local model server;
- `ollama-model-init` is a one-time initializer that pulls or creates the model;
- `api` is the FastAPI container and also the host for the `luamts` CLI command.

This means that in deployment terms:

- CLI and API are co-located in the same image;
- logically they are still separate containers in the C4 sense because they are separate applications and interaction modes;
- Ollama remains a separate runtime service.

### Internal structure narrative

#### 1. App Shell and Health Endpoint

This is intentionally thin.

Its job is to:

- create the FastAPI application object;
- configure logging once;
- register exception handlers;
- register generation routes;
- expose a health check for smoke tests and Docker validation.

This component does not know anything about Lua, Ollama prompt construction, or validation logic.

#### 2. Route Layer

The route layer provides three operational interfaces:

- `POST /generate`
- `POST /plan`
- `POST /generate/progress`

It is deliberately thin and acts as a transport adapter:

- converts request models to service kwargs;
- injects `GenerationService`;
- wraps progress events for NDJSON streaming;
- standardizes error shape through shared API error handling.

The route layer is not responsible for business decisions.

#### 3. Runtime Policy Guard

This component is central for local-only discipline.

It defines and enforces:

- `release`
- `releaseSlim`
- `debug`

It also:

- fixes compact defaults for release-like modes;
- blocks runtime overrides in release-like modes;
- rejects cloud-tagged Ollama models unless explicitly allowed in debug mode;
- ensures runtime parameters remain bounded and validated.

Architecturally, this is a policy firewall between user intent and model execution.

#### 4. Generation Orchestrator

`GenerationService` is the control tower of the backend.

It is the real center of the component architecture because it owns:

- stage ordering;
- progress emission;
- prompt assembly sequencing;
- model invocation coordination;
- validation invocation;
- semantic review trigger rules;
- bounded repair loop;
- debug payload assembly;
- temporary file cleanup for truncation guard.

In C4 terms, this component coordinates almost all other backend components.

Its main paths are:

- `plan(...)`
- `generate(...)`

Its `generate(...)` path currently implements the active flow described in docs:

`request_received -> planner -> prompter -> generation -> deterministic_validation -> semantic_validation -> response_ready`

with repair branches:

`repair_generation -> deterministic_validation -> semantic_validation`

#### 5. Planner and Clarifier Layer

This layer is split conceptually into:

- deterministic structural fallback;
- LLM-backed planner refinement;
- LLM-backed clarifier question generation.

The planner is not allowed to generate Lua.
Instead, it is responsible for shaping:

- archetype;
- operation;
- output mode;
- expected shape;
- input roots;
- risk tags;
- clarification questions.

An important implementation nuance:

- the structural `plan_task(...)` fallback initially infers roots and basic context shape;
- richer semantic choices are expected from planner-agent prompts and their parsers;
- this keeps planning small enough for constrained local model budgets.

#### 6. Prompt Builder and Prompter Layer

The prompter does not replace the whole generator prompt.

Instead, it:

- builds a strong local fallback prompt;
- optionally adds short prompt patches through an LLM-backed prompter stage;
- preserves hard constraints such as:
  - JSON-only response requirements;
  - `lua{...}lua` wrapper contract;
  - forbidden patterns like `error()`, `print`, and JsonPath.

This separation is architecturally important because it prevents prompt drift from silently changing domain rules.

Current-state nuance:

- retrieval types and assets exist in the codebase;
- `PromptBuilderResult` carries `retrieval_pack`;
- current LowCode prompt building path populates it with an empty pack, so the retrieval capability is structurally prepared but not actively enriching prompts in the primary path.

#### 7. Model Adapter

`OllamaModelAdapter` is the system's inference boundary adapter.

It encapsulates:

- local base URL enforcement;
- runtime option packaging;
- `/api/generate` calls;
- `/api/chat` calls;
- fallback to `ollama run` if needed;
- retry-on-truncation behavior for agentic prompts;
- metadata-returning prompt generation for continuation guard.

This component is intentionally the only place that knows the details of Ollama transport.

That isolation gives the rest of the system a stable abstraction:

- planner, prompter, semantic critic, and generator all become prompt producers or consumers;
- transport and runtime policy remain centralized.

#### 8. Deterministic Validation Pipeline

This subsystem is the architectural safety net of the solution.

It performs several layers of validation:

- format validation;
- syntax validation;
- static validation;
- principle validation;
- rule merging;
- selective runtime behavior validation support.

Concrete responsibilities include:

- rejecting markdown fences;
- enforcing JSON-object output where needed;
- enforcing `lua{...}lua` wrappers;
- extracting Lua segments;
- checking disallowed data roots and forbidden patterns;
- validating domain-specific failure classes;
- invoking local tooling such as `stylua`, `luacheck`, and `lua`.

This component is what makes the system more than a prompt wrapper.

#### 9. Semantic Critic

After deterministic validation passes, the system may still have a semantically wrong candidate.

The semantic critic handles that gap.

Its role is:

- compare the candidate against the task intent;
- catch semantically plausible but wrong outputs;
- return a structured semantic validator report;
- participate in final repair or finalize decisions.

This stage is deliberately placed after deterministic validation so the LLM-based semantic check does not waste budget on structurally invalid outputs.

#### 10. Critic and Repair Decision Logic

The critic report builder consolidates all reports into one repair decision.

It can choose among:

- `finalize`
- `repair`
- `clarification`

Its decision depends on:

- repairability of the primary failure;
- ambiguity of the failure;
- repair count;
- validator conflicts;
- oscillation or repeated-failure patterns;
- semantic priority rules.

This component defines the bounded agency of the system.

#### 11. Assisted Repair Summarizer

When automatic repair is exhausted, the system can still remain user-guided instead of abruptly failing.

The assisted repair summarizer:

- compresses recent failure history;
- builds a small set of user-facing repair options;
- returns an `assisted_repair_request` payload.

This is architecturally different from full autonomy:

- the system remains bounded;
- the user becomes part of the loop explicitly.

#### 12. Progress and Trace Publisher

The backend is not a black box.

It exposes:

- ordered trace stages in final payloads;
- streaming progress events for `/generate/progress`;
- debug bundles with prompt package, agent calls, model calls, and validation passes.

This observability is a key architectural property because it makes the agentic contour inspectable and demo-friendly.

### Primary internal relationships

1. Route Layer receives HTTP payloads and normalizes request kwargs.
2. Runtime Policy Guard resolves mode and runtime restrictions.
3. Generation Orchestrator calls Planner and Clarifier Layer.
4. Generation Orchestrator calls Prompt Builder and Prompter Layer.
5. Generation Orchestrator renders generator prompt and uses Model Adapter.
6. Model Adapter calls Local Ollama Runtime.
7. Generation Orchestrator sends candidate to Deterministic Validation Pipeline.
8. If deterministic validation passes, Generation Orchestrator invokes Semantic Critic through Model Adapter.
9. Critic and Repair Decision Logic determines finalize or repair.
10. If repair is needed and budget remains, Generation Orchestrator builds a repair prompt and repeats generation.
11. If repair budget is exhausted, Assisted Repair Summarizer may generate user-facing recovery options.

### Current-state implementation notes

- The active `GenerationService` flow is strongly LowCode-oriented and centered on `LOWCODE_JSON`.
- Shared domain modules support multiple output modes such as `raw_lua`, `json_wrapper`, `patch_mode`, and `clarification`.
- The broader `packages/orchestrator/repair_loop.py` module exists as a richer quality-loop library and is exercised in tests, but it is not the primary execution path behind `POST /generate`.

## Supplementary dynamic views

The user asked for a very detailed C4 description. The following textual dynamic views help explain runtime behavior without turning this file into a diagram source file.

### Dynamic view A - Main `/generate` path

1. Client sends `GenerateRequest` to `/generate` or `/generate/progress`.
2. Route layer creates or injects `GenerationService`.
3. `GenerationService` emits `request_received`.
4. `GenerationService` resolves model adapter and runtime policy.
5. Planner and optional clarifier derive task semantics and possible clarification questions.
6. Prompter builds or adjusts the generator prompt.
7. Generator prompt is rendered and sent through `OllamaModelAdapter`.
8. Candidate is collected, with continuation guard if `num_predict` truncates output.
9. Deterministic validation runs.
10. If deterministic validation succeeds, semantic critic runs.
11. Critic report decides finalize or repair.
12. If repair is required and budget remains, a repair generation pass is executed.
13. Final response returns code, status, stop reason, trace, reports, counts, and optional debug payload.

### Dynamic view B - Assisted repair path

1. Automatic repair budget is exhausted.
2. `GenerationService` does not loop indefinitely.
3. Assisted repair summarizer compresses failure history.
4. Backend returns `assisted_repair_request` with summary and options.
5. CLI or another client can re-submit the next attempt with selected assisted-repair option.

### Dynamic view C - `/plan` path

1. Client submits `PlanRequest`.
2. Backend runs planner and clarifier logic without generator or validators.
3. Response returns `task_spec`, clarification status, questions, and trace.
4. CLI can then use the returned questions to collect structured clarifications before the next generation attempt.

### Dynamic view D - CLI direct Ollama path

1. User runs `luamts generate --without-api ...`.
2. CLI builds a plain prompt locally.
3. CLI calls local Ollama directly.
4. Response is returned as `code`.
5. Validation status becomes `not_run`.

This path is architecturally useful for diagnosis, but it is not the main system value path.

## Explicit architectural constraints and non-goals

### Constraints that shape the architecture

1. Local-only inference.
2. No dependence on cloud AI APIs in normal release operation.
3. Strong domain contract around LowCode JSON and `lua{...}lua`.
4. Bounded repair instead of unbounded autonomous loops.
5. Reproducible local startup through Docker or documented local Python setup.
6. Transparent traceability through progress events and debug payloads.

### Architectural non-goals

1. Not a general-purpose coding agent.
2. Not a multi-agent platform with arbitrary tools.
3. Not an IDE-integrated autonomous internet-connected assistant.
4. Not a production-grade frontend application yet.
5. Not a fully active retrieval-augmented generation platform yet.
6. Not a persistent feedback-learning system yet.

## Current-state caveats that should appear on any C4 diagram derived from this file

1. `apps/ui` is a placeholder only.
2. `packages/model-adapter` is also a placeholder only; the active model adapter lives in `apps/api/adapters/model.py`.
3. `knowledge/feedback/` is future-facing and not wired into runtime.
4. Retrieval infrastructure exists, but the main LowCode prompt builder currently uses an empty `RetrievalPack`.
5. `packages/orchestrator/repair_loop.py` is a richer quality-loop module, but `GenerationService` implements the primary active generation and repair path behind the API.
6. The main runtime path is strongly specialized for LowCode JSON output, even though shared modules recognize multiple output modes.

## Concise C4 summary

If reduced to one paragraph:

`luaMTS` is a local-first LowCode Lua generation system whose core runtime consists of a CLI and a FastAPI backend that orchestrate planner, prompter, generator, deterministic validators, semantic critic, and bounded repair logic over a local Ollama model runtime, while relying on repository-local prompt assets, datasets, benchmark scripts, and report storage to keep the solution reproducible, inspectable, and domain-constrained.
